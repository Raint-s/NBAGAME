# 初始代码审计记录

日期：2026-06-02

## 审计范围

本次审计聚焦于当前仓库的上线前风险，而不是玩法优化。

覆盖范围：

- `game.js`
- `index.html`
- `sw.js`
- `manifest.webmanifest`
- `capacitor.config.json`
- `docs/platforms/android-packaging.zh-CN.md`（原 `README_APK.md`）

## 审计结论

当前项目适合作为原型继续开发，但还不适合作为带真实账号、真实支付、真实联网能力的公开发布版本直接上线。

## 主要问题

### 高风险 1：账号、钻石、支付结果完全由前端本地控制

现状：

- 账号数据通过 `localStorage` 保存
- 当前用户、钻石余额、支付结果均由前端直接写入

相关位置：

- `game.js` 中账户与本地存储逻辑
- `game.js` 中充值、支付完成逻辑

影响：

- 任何人都可以在浏览器中直接修改账号或钻石数据
- 不具备真实账号体系或真实付费体系所需的基本可信度

建议：

- 原型期继续保留可以
- 公开上线前必须拆分为真实后端鉴权 / 订单 / 校验链路
- 如果短期不接真实能力，至少要在 UI 与文案层明确标注为演示功能并限制入口

### 高风险 2：微信 / QQ 登录与支付方式属于前端模拟流程

现状：

- 使用真实平台名称展示授权和支付流程
- 实际只是在前端等待一段时间后本地完成

影响：

- 在公开版本中容易造成误解
- 对外发布时存在产品与合规风险

建议：

- 原型演示环境可保留
- 正式发布前至少默认关闭这些入口，或清晰标记为“演示功能”

### 中风险 1：存档保存不完整，读档后会丢状态

现状：

- 团队对象含有 `stats`、`claimed`、`boostUntil`、`lastMatchAt` 等状态
- 存档只保存了其中一部分

影响：

- 读档后任务进度、奖励领取状态、冷却、增益时间等可能失真

建议：

- 在下一轮修复中统一存档模型
- 明确哪些字段必须持久化，哪些字段可重建

### 中风险 2：service worker 缓存策略过宽

现状：

- 拦截全部 `GET`
- 默认缓存网络响应
- 失败时直接回退到 `index.html`

影响：

- 将来一旦接入后端、远程配置、版本切换，排查问题会很麻烦

建议：

- 下一轮按静态资源与接口请求分开处理缓存策略

### 中风险 3：Android mixed content 当前开启

现状：

- `capacitor.config.json` 中 `allowMixedContent: true`

影响：

- 原型阶段方便
- 公开发布前属于应重点复查的放宽项

建议：

- 在发布前确认是否仍然需要
- 若无必要，关闭

## 修复优先级建议

1. 先处理模拟登录 / 模拟支付的公开入口策略
2. 再修存档模型
3. 然后收紧 service worker 策略和移动端配置
